What’s New in This Release of the VBA SDK?

The following items are have been updated for the VBA 6.0 Software Development Kit (SDK) version 6.5:

·         Updated version of VBA

·         Updated version of the Application Programmability Component (APC)

·         Updated APC merge modules

·         Updated VBA Core Components installer—VBAOF11.msi

·       MSO 11

·         Support for Visual Studio .NET

 

Fixes included in VBA 6.5 (vbe6.dll)

·         Fixes for instabilities reported through Windows Error Reporting

·         Fixes specifically designed to enhance the stability and security of VBE

·         Sendkeys() support on Windows Vista

o   Sendkeys calls fail on Vista due to code access permission changes in the OS.

·         More robust handling of documents that contain many ActiveX controls.

o   Documents with large numbers (hundreds) of ActiveX controls could crash or hang the host application.

·         IDispatch::Invoke does not fire events when the application is in a modal state

o   An example would be when the host application displays a modal progress dialog while VBA code is executing. This situation does not occur when using ITypeInfo::Invoke.

·         Mouse wheel support fixed in the VBA code window

The Updated APC

All host applications should use the version of APC in this release. The following list highlights some of the changes made to APC for this release:

·         Security enhancements.

·         MSO 11 includes security and stability enhancements.

·         VBA 6.5 will support only the newer version of MSO.

Because the preceding changes represent risks to existing applications, a new version of APC, APC65.dll, was created. This DLL can coexist with previous versions of APC, ensuring compatibility with current applications. The APC.msm has been updated to include the APC65.dll. We recommend that you upgrade your applications to include the new APC in order to benefit from current fixes made to the APC as well as from fixes included in previous versions.

·         If you are upgrading your application to take advantage of the new APC, you must modify your application to use APC65.dll. This will be done differently depending on how your application was developed:

·         Applications that use the MFC or ATL wrappers should only require you to recompile using the new wrappers included with the VBA 6.0 SDK version 6.5.

·         Applications developed in Visual Basic need to have the project reference changed from the former version of APC to APC65.dll.

Fixes included in APC 6.5 (apc65.dll)

·         Some applications unable to open document when VBE and APC versions are mismatched.

·         Some applications fail during .EXD (extended type library) file creation on Vista.

·         MFC71 and APCMFC.H header files updated to resolve the following build issues in Visual Studio 7.0, 7.1 and 8.0.

1.       Error when building Unicode application error generates the following error.

§  c:\program files\vba6sdk\include\apcmfc.h(381) : error C2065: 'm_nDisablePumpCount' : undeclared identifier
c:\program files\vba6sdk\include\apcmfc.h(357) : while compiling class template member function 'BOOL MSAPC::CApcApplication<T>::PumpMessage(void)'

2.       VBA IDE not operating correctly when in a modal state. Event processing can hang if significant time passes between OnIdle == True setting and the subsequent WaitMessage() call.

Installation Instructions

Preparing Your Computer

If you installed previous releases of VBA 6.0 components and the VBA 6.0 SDK, you must uninstall these before installing the release software. Microsoft does not support installation on top of previous releases of the VBA SDK.

Uninstalling Previous Releases

Before installing the release version of the VBA SDK, you must uninstall any previous releases.

To uninstall previous versions of the VBA 6.0 SDK

Use Add/Remove Programs in Control Panel to uninstall the VBA SDK.

System Requirements

The VBA SDK contains information for integrating VBA version 6.5 with host applications that operate on Intel-based platforms running any of the following operating systems: Windows 2000, Windows XP, or Windows Server 2003. This version of the VBA SDK does not run on Windows NT Workstation version 3.5x, Windows NT Workstation version 4.0, Windows 95, Windows 98, or Windows Millennium Edition.

Note It is recommended that you install the VBA 6.0 SDK on released operating systems only.

A typical installation, which contains only the English version of VBA, requires approximately 130 MB of hard disk space; a complete installation, which includes all languages, requires approximately 310 MB of hard disk space.

Application Requirements

It is recommended that you install Microsoft Visual Basic 6.0 SP6 or later to host VBA in Visual Basic. The Visual Basic Integration Wizard will be installed by the SDK setup only if Visual Basic 6.0 SP3 or later is present on the installation computer.

Installing VBA 6.0 SDK

1.       Quit all programs.

2.       Run Setup.exe from the root directory of the VBA SDK CD.

3.       Click Next.

4.       Accept the license agreement and choose the desired components from subsequent dialog boxes.

5.       After all selections have been made, click Install.

Switching Between Release and Debug VBA Components

1.       Quit all programs.

2.       Run Add/Remove Programs from the Control Panel.

3.       Click Next in the Welcome dialog box.

4.       Click Modify in the Program Maintenance dialog box.

5.       Click Next in the Custom Setup dialog box.

6.       Choose the release or debug version and click Next in the version selection dialog box.

7.       Click Install.

8.       After the SDK installation is complete, click Finish. At this point, the release or debug VBA core component will be installed.

Note The debug version of VBA components does not coexist with versions of Microsoft Office. You must install retail VBA components to run these installations of Microsoft Office.

Building and Running the Sample Host Applications

To build any of the samples, you must install and register the VBA components. The VBACalc sample (executable) and the MultiPad sample must be built in Microsoft Visual C++ version 5.0 SP3 or later. The SDINote sample must be built in Microsoft Visual Basic version 6.0 SP3 or later.

Viewing Help

Help for the VBA SDK is in HTML Help format. All Help for the SDK is included in one .chm file, vba6sdk.chm.

To access Help

1.       Choose Programs from the Windows Start menu.

2.       Choose SDK Documentation from the VBA 6.0 SDK menu item.

3.       Click the SDK Reference command.

To navigate through the Help file, use the Contents tab, Index tab, or Search tab.

Known Issues

VBA Issues

1.       The VBA IDE may attempt to default to English if the system locale and installed application localized resources do not match resulting in a VBA initialization error. To protect against this possible condition, always include the English redist component with your application setup.

2.       If your development environment is on, or if your application will distribute to, Traditional Chinese Windows 98/NT 4.0, please contact Microsoft Taiwan Corporation Product Support Service for instructions on updating the system file for the Taiwan calendar function.

3.       The number of projects you can create simultaneously depends on the number of project items in the project. VBA allocates a 64-KB block of memory for each project.

4.       When updating compatible type libraries, you must place new additions, including typedef statements, at the end of the .odl or .idl script because VBA requires that the offsets to existing type information remain unchanged. Otherwise, errors will result when you try to run a project that uses a previous version of the .tlb file on a host with the current .tlb file. Should this occur, the only solution is to force the project to recompile (for example, by adding a new procedure to the project).

5.       The digital signature feature of VBA 6.0 requires that Internet Explorer version 4.0 or later be installed. If your application enables digital signature support and a supported version of Internet Explorer is not available, the digital signatures feature of VBA 6.0 will be disabled. Makecert.exe and Selfcert.exe also require Internet Explorer version 4.0 or later. Selfcert.exe is used to create test signatures to test the digital signature feature of VBA.

6.       The Visual Basic Editor defaults to English if the current system code page does not support the user interface language passed during initialization. For this reason, you are advised to always install the English components in addition to other desired languages.

7.       If you are using WinHelp, pressing F1 to access Help might not work from within a UserForm in the Visual Basic Editor. Use the Help menu commands directly to locate the desired topic.

8.       If you are using the host add-in support provided by VBA, you should initialize the IDE before asking VBA to load the add-ins. Otherwise, the end user might encounter a run-time error if a modeless UserForm is displayed in the add-in OnConnect method.

Caution Showing a modeless form in a multithreaded project will result in potentially unrecoverable errors.

1.       The multithread designer does not display or produce an error if the type library for the appobject cannot be loaded. Ensure that the Locale Identifier (LCID) for the type library matches the LCID used to initialize VBA. The best way to do this is to set the type library LCID to 0, or to language neutral.

Caution Passing ProgIDs longer than 256 characters to IApcControls::AddByProgID, or other, similar methods, will result in potentially unrecoverable errors.

2.       When the host application is able to load a document but does not, and does not want the document opened as a type library, it should return VBA_E_UNSUPFORMAT, not E_FAIL, when returning from IVbaSite::OpenProject.

3.       If your application supports both controls on document project items and interoperability of documents with a previous version of your application, it is recommended that the type information of your extender stay the same. If you must change the extender type information, then the change should be binary compatible. Even if your extender type information does not change, the prepend path or directory passed to IExtendedTypeLib::AddRefExtendedTypeLib or to AddRefExtendedTypeLibOfCLSID must remain the same between versions. If your extender type information does change, upon upgrade to your new version it is recommended that your extended controls LIB directory be cleaned out to force the creation of new extended .lib files based on your new extender.

4.       The online Help for Swedish VBA will be in English.

5.       The implementation of the VBA extender's Object property exhibits behavior that potentially can produce unrecoverable errors. For dual interface controls extended by VBA, the Object property, as implemented by the VBA extender, does not return the control's full dual interface when queried to do so. The Object property returns only an IDispatch implementation, which can produce unrecoverable errors if the caller assumes that a full dual interface is returned and subsequently calls directly into the custom portion of the returned interface pointer.

6.       The following describes additional properties that apply to class modules in stand-alone projects:

Instancing Property

Sets a value specifying whether you can create instances of a public class outside a project and, if so, how they will behave. This property is not available at run time.

The Instancing property has these settings:

(Default) Private. Other applications are not allowed access to type library information about the class and cannot create instances of the class. Private objects are for use only within your component. The Instancing property default varies depending on the project type. Private is the default only for class modules in stand-alone projects.

 PublicNotCreatable. Other applications can use objects of this class only if your component creates the objects first. Other applications cannot use the CreateObject function or the New operator to create objects from the class.

Name Property

Returns the name used in code to identify a class. It is read-only at run time.

Syntax

object.Name

The object variable represents an object expression that evaluates to a class.

Remarks

An object's Name property must start with a letter and can be a maximum of 40 characters. The Name property can include numbers and underscore (_) characters but cannot include punctuation or spaces. Although the Name property setting can be a keyword, property name, or the name of another object, this can create conflicts in your code.

7.       Hosts should not call the IVBAProcs::GetVbaProcs method during idle time when VBA is the active component.

8.       In order for hidden projects to remain hidden, do not use Autofill in the Macros dialog box or add hidden projects to a scope.

9.       Hosts should not call a control that is being actively debugged in Visual Basic.

10.   Visual Basic sometimes fails when running VBA-enabled applications that use digital signatures. This failure can occur when the VBA IDE displays the Digital Signature dialog box during the debugging of a VBA-enabled application in Visual Basic.

11.   The IVBA::LoadMTDll method fails when the path to the DLL exceeds 144 characters.

12.   Calling a disabled Publish menu item with the VBE Execute method will return E_FAIL and may cause the application to fail.

13.   An assertion may appear when the VBA IDE is first displayed from an application when using debug VBA components:

file C:\MSO9\tcutil.cpp line 6180: could not fix LanguageResources registry entries

This assertion does not affect the operation of VBA and may safely be ignored. The assertion does not appear in the release VBA components.

APC Issues

No new issues to report with version 6.5

1.       When running the debug version of the VBA SDK samples, two assertions appear during execution:

·         file d:\office11\dev\mso\sqm.cpp line 5744: TAG 'ydck' SQM: Systray UI without ever registering SQM??? 0x0

·         file d:\office11\dev\mso\sqmconfig.cpp line 1273: TAG 'ydcl' SQM: Using SQM without registering SQM (yet) ??? 0x0

These assertions have no apparent effect on execution of the VBA samples and can safely be ignored.

2.       Project items cannot be implemented out of process because they are aggregated with extender objects and COM does not support this type of aggregation. If this is attempted, APC may fail.

3.       Using the IVbaProjectSite::PutSignature method to remove a Digital Certificate from a VBA Project does not mark that project as modified.

4.       IApcProcedures collection methods return errors when used on a locked project.

5.       A new class, CApcOleServerDoc, has been added to the MFC integration layer of APC (ApcMfc.h). This class addresses potential persistence problems that can occur when embedding CApcDocument derived objects in certain ActiveX containers, including Word97 and Excel97. The new class overrides some of the CApcDocument methods as well as MFC's default implementation of IPersistStorage in COleServerDoc. To take advantage of these changes, it is recommended that you use CApcOleServerDoc as a base class for your embeddable (COleServerDoc based) documents.

Note ApcOleServerDoc relies on the CApcDocument class and should use it as a base, which in turn requires a COleServerDoc base.

Reading Files from Previous Versions of the Host

Hosts should bump the minor version of the host executable only when the associated type library is updated and if the new type library remains compatible with older files. If the minor version is not bumped, the major version of the executable should be bumped.

VBA Installer Issues

Note: The VBA Installer has been deprecated. Host application setup development can now be accomplished using the latest versions of InstallShield and Wise. This change, along with the new VBA Core components MSI, allows better application installation, maintenance, and uninstallation performance by taking full advantage of Windows Installer technologies.

VS Installer Issues

In order for a VB application to run properly in the Windows 2000 Restricted User mode, an Administrator must first run the application. This is because the VS Installer cannot get the registry information required to properly register the VB classes in a project during installation. The registry entries may also be manually inserted into the VS Installer, as detailed in the Windows 2000 Considerations topic in the VBA SDK 6.0 version 6.5 Help.

VBA Merge Module Issues

1.       The VBA merge modules work with the following installation tools:

·         Visual Studio Installer 7.0 or later

·         InstallShield for Windows Installer version 8.0 or later

·         Wise for Windows Installer version 5.0 or later

·         Windows Installer 2.0 or later

2.       Visual Studio Installer 7.0 reports warnings when some VBA merge modules are added to the setup project. These warnings can be ignored.

3.       The merge modules shipped in the VBA 6.0 SDK version 6.5 are different from those shipped in version 6.1 and version 6.2 of the SDK. For example, there is a new APC6.5 component in APC.msm. For this reason, the installation package should not use the same product code used for previous versions of your product. See the Windows Installer documentation for more information.

Setup Issues

Previously installed VBA Core components (vbaoff11.msi) must be uninstalled before new version can be installed. The following can be include of your setup to uninstall the previously installed VBA Core.

miexec.exe /x vbaof11.msi

Installing the SDK and HTML Help on Windows NT version 4.0 requires administrator privileges. Installing on Windows 2000 requires Administrator or Power User privileges.

Sample Application Issues

1.       "Access Denied" when attempting to load sample game file in Spruuids sample. Sample game files found in \VBA6SDK\samples\ATL-C++\Spruuids\Data are installed as read only causing an Access Denied error when opening the file. Remove the read-only attribute to resolve the error and allow loading of sample game files.

2.       The files Makecert.exe and Selfcert.exe are not installed with the samples. Rather, they are installed to the \Program Files\Microsoft Office\Office directory. Both files are installed with the VBA setup, not with the SDK setup. Makecert.exe creates test signatures for testing the digital signature feature of VBA. Selfcert.exe is the user interface for the Makecert program.

3.       The debug versions of the new MFC DLLs released with Visual Studio 6.0 SP1 and later are not guaranteed to be binary compatible with previous debug versions. Therefore, the debug versions of MultiPad.exe and VBACalc.exe have been linked with the retail versions of the MFC DLLs. If you recompile the sample to link to the debug versions of the MFC DLLs, the resulting binaries may not run on systems with incompatible versions of MFC.

4.       All samples use HTML Help by default.

5.       All samples set the VBA IDE to English by default. To have the samples use another language, run the sample executable from the command line with the /VBALang=LangNum flag, where LangNum equals the decimal value for the LCID of the desired language.

6.       VBACalc Sample: Events may fail to fire if the Document project item does not have a name property.

Directory Structure

\vba6sdk

Contains all VBA SDK files. Included in the root are the Help files and installation files for the VBA SDK.

\vba6sdk\Include

Contains the files that must be included in your host application during integration development. Add the \Include directory path to your compiler’s include search path.
